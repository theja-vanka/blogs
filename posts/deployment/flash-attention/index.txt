<section id="introduction" class="level2">
<h2 class="anchored" data-anchor-id="introduction" id="introduction">1. Introduction</h2>
<p>Attention is the computational core of nearly every modern image model — Vision Transformers (ViT) for classification and representation learning, Diffusion Transformers (DiT) and U-Net-based diffusion models for image generation, and multimodal models like CLIP that align vision and language. Attention’s expressive power comes at a steep price: the standard implementation scales quadratically in both compute and memory with the number of tokens. For images, “tokens” usually means patches, and high-resolution images or long video/image sequences can quickly produce tens of thousands of tokens, making naive attention the dominant bottleneck in both training and inference.</p>
<p>FlashAttention is a family of exact (non-approximate) attention algorithms designed to close the gap between the theoretical FLOP count of attention and its real-world wall-clock performance. It does this not by changing the math of attention, but by changing <em>how</em> the computation is scheduled on the GPU memory hierarchy. The result is the same output as standard attention, computed several times faster and with dramatically less memory, which in turn allows vision models to use longer sequences (higher resolutions, more patches, longer video clips) without running out of memory or grinding to a halt.</p>
<p>This guide walks through why attention is slow, how FlashAttention fixes it, how the algorithm has evolved across FlashAttention-1, -2, and -3, and — most importantly for this guide’s focus — how all of this applies specifically to image models: ViTs, diffusion U-Nets and DiTs, and vision-language models. It closes with practical code, library options, benchmarks, and a discussion of caveats and limitations.</p>
</section>
<section id="why-standard-attention-is-slow-a-memory-problem-not-a-compute-problem" class="level2">
<h2 class="anchored" data-anchor-id="why-standard-attention-is-slow-a-memory-problem-not-a-compute-problem" id="why-standard-attention-is-slow-a-memory-problem-not-a-compute-problem">2. Why Standard Attention Is Slow: A Memory Problem, Not a Compute Problem</h2>
<p>The scaled dot-product attention formula is simple:</p>
<p><span class="math display">\[
\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d}}\right) V
\]</span></p>
<p>Given query, key, and value matrices <span class="math inline">\(Q, K, V \in \mathbb{R}^{N \times d}\)</span>, where <span class="math inline">\(N\)</span> is the sequence length (number of image patches, for a ViT) and <span class="math inline">\(d\)</span> is the head dimension, the naive implementation does the following:</p>
<ol type="1">
<li>Compute <span class="math inline">\(S = QK^\top\)</span>, an <span class="math inline">\(N \times N\)</span> matrix. This requires <span class="math inline">\(O(N^2d)\)</span> FLOPs and, critically, <span class="math inline">\(O(N^2)\)</span> memory to store <span class="math inline">\(S\)</span>.</li>
<li>Apply softmax row-wise to <span class="math inline">\(S\)</span> to get the attention probability matrix <span class="math inline">\(P\)</span> (also <span class="math inline">\(N \times N\)</span>, also materialized in memory).</li>
<li>Compute <span class="math inline">\(O = PV\)</span>, another <span class="math inline">\(O(N^2d)\)</span> FLOP operation.</li>
</ol>
<p>The FLOP count here is not actually the problem on modern GPUs — matrix multiplications are extremely well optimized and GPUs have enormous FLOP throughput. The real problem is that steps 1 and 2 require writing and reading the full <span class="math inline">\(N \times N\)</span> matrices <span class="math inline">\(S\)</span> and <span class="math inline">\(P\)</span> to and from <strong>high-bandwidth memory (HBM)</strong>, the GPU’s main memory (what you’d call VRAM). HBM has large capacity but relatively limited bandwidth compared to the GPU’s on-chip <strong>SRAM</strong> (shared memory / registers), which is extremely fast but tiny (on the order of 100–200 KB per streaming multiprocessor).</p>
<p>Standard attention is therefore <strong>memory-bandwidth bound</strong>, not compute bound: the GPU’s arithmetic units sit idle much of the time, waiting for data to be shuttled between HBM and SRAM. This is exactly the kind of inefficiency that IO-aware algorithm design can attack, and it’s exactly what FlashAttention does.</p>
<p>For image models, this quadratic-in-<span class="math inline">\(N\)</span> memory and bandwidth cost is especially painful. A ViT processing a <span class="math inline">\(224 \times 224\)</span> image with <span class="math inline">\(16 \times 16\)</span> patches has <span class="math inline">\(N = 196\)</span> tokens — manageable. But a ViT-style backbone or DiT operating on a <span class="math inline">\(1024 \times 1024\)</span> image with the same patch size has <span class="math inline">\(N = 4096\)</span> tokens, and video-diffusion or multi-frame vision models can push <span class="math inline">\(N\)</span> into the tens of thousands. Since the attention matrix grows as <span class="math inline">\(N^2\)</span>, doubling resolution (which roughly quadruples patch count) can increase attention memory by <span class="math inline">\(16\times\)</span>. This is precisely why high-resolution image and video generation models were historically constrained to low resolutions or required aggressive downsampling/windowing before FlashAttention-style kernels became standard.</p>
</section>
<section id="the-core-idea-io-awareness-via-tiling-and-online-softmax" class="level2">
<h2 class="anchored" data-anchor-id="the-core-idea-io-awareness-via-tiling-and-online-softmax" id="the-core-idea-io-awareness-via-tiling-and-online-softmax">3. The Core Idea: IO-Awareness via Tiling and Online Softmax</h2>
<p>FlashAttention (Dao et al., 2022) reframes attention as an IO-aware algorithm: instead of asking “how do I minimize FLOPs,” it asks “how do I minimize the number of reads and writes to slow HBM.” Two techniques make this possible.</p>
<section id="tiling" class="level3">
<h3 class="anchored" data-anchor-id="tiling" id="tiling">3.1 Tiling</h3>
<p>Rather than computing the full <span class="math inline">\(S = QK^\top\)</span> matrix at once, FlashAttention splits <span class="math inline">\(Q\)</span>, <span class="math inline">\(K\)</span>, and <span class="math inline">\(V\)</span> into blocks small enough to fit into fast on-chip SRAM. For each pair of blocks (a block of <span class="math inline">\(Q\)</span> and a block of <span class="math inline">\(K/V\)</span>), it:</p>
<ol type="1">
<li>Loads the <span class="math inline">\(Q\)</span>-block and <span class="math inline">\(K/V\)</span>-block from HBM into SRAM once.</li>
<li>Computes the partial attention scores for just that block, entirely within SRAM.</li>
<li>Accumulates a running (partial) output and running softmax statistics.</li>
<li>Moves to the next <span class="math inline">\(K/V\)</span> block, repeating, without ever writing the full <span class="math inline">\(N \times N\)</span> score or probability matrix back to HBM.</li>
</ol>
<p>This means the intermediate <span class="math inline">\(N \times N\)</span> matrices <span class="math inline">\(S\)</span> and <span class="math inline">\(P\)</span> are never materialized in HBM at all — they exist only transiently, block by block, inside fast on-chip memory. This is the single biggest lever: FlashAttention reduces HBM accesses from</p>
<p><span class="math display">\[
\Theta(Nd + N^2)
\]</span></p>
<p>in the standard algorithm to roughly</p>
<p><span class="math display">\[
O\!\left(\frac{N^2 d^2}{M}\right)
\]</span></p>
<p>where <span class="math inline">\(M\)</span> is the SRAM size — for typical head dimensions and SRAM sizes, this works out to roughly an order of magnitude (often cited around <span class="math inline">\(5\)</span>–<span class="math inline">\(25\times\)</span>) fewer memory accesses, which translates directly into wall-clock speedups even though the FLOP count is essentially unchanged.</p>
</section>
<section id="online-streaming-softmax" class="level3">
<h3 class="anchored" data-anchor-id="online-streaming-softmax" id="online-streaming-softmax">3.2 Online (Streaming) Softmax</h3>
<p>Tiling raises an obvious problem: softmax needs to normalize across the <em>entire</em> row of S (i.e., across all keys), but tiling only ever sees one block of keys at a time. The trick FlashAttention borrows and formalizes is the <strong>online softmax</strong> algorithm: instead of computing the softmax normalization constant in one pass over the whole row, it maintains a running maximum and running sum of exponentials as each new K/V block is processed, and rescales the accumulated output whenever the running maximum changes. This produces mathematically identical results to a full-row softmax while only ever needing to see one block of keys at a time, which is what makes single-pass, block-by-block computation possible without approximation.</p>
</section>
<section id="recomputation-in-the-backward-pass" class="level3">
<h3 class="anchored" data-anchor-id="recomputation-in-the-backward-pass" id="recomputation-in-the-backward-pass">3.3 Recomputation in the Backward Pass</h3>
<p>Standard attention implementations save the <span class="math inline">\(N \times N\)</span> attention matrix <span class="math inline">\(P\)</span> from the forward pass so it can be reused during backpropagation to compute gradients. FlashAttention deliberately does <em>not</em> store <span class="math inline">\(P\)</span>. Instead, during the backward pass, it <strong>recomputes</strong> the needed blocks of <span class="math inline">\(S\)</span> and <span class="math inline">\(P\)</span> on the fly from <span class="math inline">\(Q\)</span>, <span class="math inline">\(K\)</span>, <span class="math inline">\(V\)</span> and the small saved softmax statistics (the running max <span class="math inline">\(m\)</span> and sum <span class="math inline">\(\ell\)</span> per row), using the same fast SRAM-resident tiling strategy. This trades a modest amount of extra compute (recomputation) for a large reduction in memory reads/writes, and turns out to be a net win because HBM bandwidth, not FLOPs, is the bottleneck. This is the same “recomputation” idea used in gradient checkpointing, applied specifically to the attention matrix.</p>
<p><em>(See the equations in Section 4 for the exact form of <span class="math inline">\(S\)</span>, <span class="math inline">\(P\)</span>, <span class="math inline">\(m\)</span>, and <span class="math inline">\(\ell\)</span> that get recomputed block-by-block.)</em></p>
</section>
<section id="net-effect" class="level3">
<h3 class="anchored" data-anchor-id="net-effect" id="net-effect">3.4 Net Effect</h3>
<p>The combination of tiling and online softmax means FlashAttention computes <strong>mathematically exact</strong> attention (no approximation, unlike sparse or low-rank attention variants) while reducing memory usage from <span class="math inline">\(O(N^2)\)</span> to <span class="math inline">\(O(N)\)</span> and reducing wall-clock latency by multiples (commonly <span class="math inline">\(2\)</span>–<span class="math inline">\(4\times\)</span> in the original paper, more in later versions) due to far fewer HBM round trips. This is why FlashAttention was adopted almost universally rather than being treated as one option among many approximate-attention schemes: it changes <em>how</em> attention is computed, not <em>what</em> it computes.</p>
</section>
</section>
<section id="algorithm-walkthrough" class="level2">
<h2 class="anchored" data-anchor-id="algorithm-walkthrough" id="algorithm-walkthrough">4. Algorithm Walkthrough</h2>
<p>The following walkthrough captures the forward pass structure. Assume <span class="math inline">\(Q, K, V \in \mathbb{R}^{N \times d}\)</span> live in HBM, and are split into blocks of size <span class="math inline">\(B_r\)</span> (query rows) and <span class="math inline">\(B_c\)</span> (key/value rows) chosen so that each block fits in SRAM. The output accumulator <span class="math inline">\(O \in \mathbb{R}^{N \times d}\)</span> and the per-row softmax statistics <span class="math inline">\(\ell \in \mathbb{R}^N\)</span> (running sum of exponentials) and <span class="math inline">\(m \in \mathbb{R}^N\)</span> (running row maximum) are initialized to zero, zero, and <span class="math inline">\(-\infty\)</span> respectively, and live in HBM.</p>
<p><strong>Outer loop</strong>, over query blocks <span class="math inline">\(Q_i\)</span> (loaded once into SRAM per iteration, with local accumulators <span class="math inline">\(O_i = 0\)</span>, <span class="math inline">\(\ell_i = 0\)</span>, <span class="math inline">\(m_i = -\infty\)</span>):</p>
<p><strong>Inner loop</strong>, over key/value blocks <span class="math inline">\(K_j, V_j\)</span> (loaded into SRAM), updates the running statistics with the following equations:</p>
<p><span class="math display">\[
\begin{aligned}
S_{ij} &amp;= \frac{Q_i K_j^\top}{\sqrt{d}} \\[4pt]
m_i^{\text{new}} &amp;= \max\!\left(m_i,\ \operatorname{rowmax}(S_{ij})\right) \\[4pt]
P_{ij} &amp;= \exp\!\left(S_{ij} - m_i^{\text{new}}\right) \\[4pt]
\ell_i^{\text{new}} &amp;= e^{\,m_i - m_i^{\text{new}}}\, \ell_i + \operatorname{rowsum}(P_{ij}) \\[4pt]
O_i &amp;\leftarrow e^{\,m_i - m_i^{\text{new}}}\, O_i + P_{ij} V_j \\[4pt]
m_i, \ell_i &amp;\leftarrow m_i^{\text{new}}, \ell_i^{\text{new}}
\end{aligned}
\]</span></p>
<p>Here <span class="math inline">\(S_{ij}\)</span> is the block of raw attention scores and <span class="math inline">\(P_{ij}\)</span> the (unnormalized) block of attention probabilities — both live only transiently in SRAM. The exponential rescaling terms <span class="math inline">\(e^{\,m_i - m_i^{\text{new}}}\)</span> are exactly the online-softmax correction: whenever a new key/value block raises the running row maximum, the previously accumulated output and normalizer are rescaled to stay consistent, so the final result is identical to a full-row softmax.</p>
<p>After the inner loop finishes for a given query block, the accumulator is normalized and written back to HBM:</p>
<p><span class="math display">\[
O_i \leftarrow \frac{O_i}{\ell_i}
\]</span></p>
<p>with <span class="math inline">\(O_i\)</span> written to HBM as the final output, and the small statistics <span class="math inline">\(m_i, \ell_i\)</span> also written to HBM (used later for backward-pass recomputation).</p>
<p>Notice that <span class="math inline">\(S_{ij}\)</span> and <span class="math inline">\(P_{ij}\)</span> — the expensive <span class="math inline">\(N \times N\)</span>-shaped quantities — never leave SRAM and are never stored in full. Only the final output <span class="math inline">\(O\)</span> (size <span class="math inline">\(N \times d\)</span>) and two small per-row statistics vectors (<span class="math inline">\(m\)</span> and <span class="math inline">\(\ell\)</span>, each size <span class="math inline">\(N\)</span>) are ever written back to HBM. This is the entirety of the “IO-awareness” that gives the algorithm its name.</p>
</section>
<section id="from-flashattention-1-to-flashattention-3" class="level2">
<h2 class="anchored" data-anchor-id="from-flashattention-1-to-flashattention-3" id="from-flashattention-1-to-flashattention-3">5. From FlashAttention-1 to FlashAttention-3</h2>
<section id="flashattention-v1" class="level3">
<h3 class="anchored" data-anchor-id="flashattention-v1" id="flashattention-v1">5.1 FlashAttention (v1)</h3>
<p>Introduced by Tri Dao et al.&nbsp;in “FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness” (2022), this established the tiling + online-softmax + recomputation approach described above. It delivered exact attention with linear memory in sequence length and substantial wall-clock speedups (commonly cited as 2–4× over standard PyTorch attention, and larger gains at longer sequence lengths), immediately making longer-context training and higher-resolution vision workloads more tractable.</p>
</section>
<section id="flashattention-2" class="level3">
<h3 class="anchored" data-anchor-id="flashattention-2" id="flashattention-2">5.2 FlashAttention-2</h3>
<p>FlashAttention-2 (Dao, 2023) kept the same IO-aware foundation but re-engineered the <em>work partitioning and parallelism</em> to better use GPU hardware:</p>
<ul>
<li><strong>Reduced non-matmul FLOPs.</strong> The original algorithm spent relatively more time on softmax rescaling operations, which don’t map efficiently onto GPU tensor cores. FA2 restructured the inner loop to minimize this non-matmul overhead.</li>
<li><strong>Better parallelism across thread blocks.</strong> FA1 parallelized primarily over batch size and number of heads, which underutilizes the GPU when batch×heads is small relative to the number of streaming multiprocessors (a common situation for long-sequence workloads, exactly the regime high-resolution vision models push into). FA2 also parallelizes over the sequence dimension, keeping the GPU busier even at small batch sizes.</li>
<li><strong>Better work partitioning within a thread block</strong>, reducing communication between warps.</li>
</ul>
<p>Net result: FlashAttention-2 is reported at roughly 1.7–3.0× faster than FlashAttention-1 (and 1.3–2.5× faster than a Triton implementation of FA1), reaching 50–73% of theoretical peak FLOPs/s on A100 GPUs, versus 3–10× faster than a standard (non-fused) attention implementation. This made FA2 the default choice integrated into PyTorch’s <code>scaled_dot_product_attention</code>, Hugging Face Transformers/Diffusers, and xFormers.</p>
</section>
<section id="flashattention-3" class="level3">
<h3 class="anchored" data-anchor-id="flashattention-3" id="flashattention-3">5.3 FlashAttention-3</h3>
<p>FlashAttention-3 (Shah, Bikshandi, Dao et al., 2024) targets NVIDIA Hopper GPUs (H100/H800) specifically, exploiting hardware features not available on older architectures:</p>
<ul>
<li><strong>Warp specialization and asynchrony.</strong> FA3 overlaps the (asynchronous) memory-copy and matmul instructions with the (synchronous) softmax computation, using Hopper’s asynchronous Tensor Memory Accelerator (TMA) and warp-group matrix multiply instructions (WGMMA) so that data movement and floating-point work happen concurrently rather than serially.</li>
<li><strong>TMA offload.</strong> The Tensor Memory Accelerator handles HBM↔︎SRAM data movement and index/bounds calculations in hardware, freeing up GPU registers that can then be used to increase tile size and further improve efficiency.</li>
<li><strong>Low-precision (FP8) support.</strong> FA3 adds block quantization and incoherent processing techniques so that attention can run in FP8 with much smaller accuracy loss than a naive FP8 cast would produce.</li>
</ul>
<p>FlashAttention-3 requires an H100/H800-class GPU and CUDA ≥ 12.3 (12.8 recommended). Reported results include roughly 1.5–2.0× speedup over FlashAttention-2 on H100 in FP16, reaching up to ~740 TFLOPs/s (about 75% of the H100’s theoretical FP16 peak), and close to 1.2 PFLOPs/s when using FP8. FlashAttention-2 remains the appropriate choice on Ampere/Ada GPUs (A100, RTX 30/40-series), since FA3’s Hopper-specific hardware paths simply aren’t available there.</p>

<section id="why-this-matters-specifically-for-image-models" class="level2">
<h2 class="anchored" data-anchor-id="why-this-matters-specifically-for-image-models" id="why-this-matters-specifically-for-image-models">6. Why This Matters Specifically for Image Models</h2>
<section id="vision-transformers-vit" class="level3">
<h3 class="anchored" data-anchor-id="vision-transformers-vit" id="vision-transformers-vit">6.1 Vision Transformers (ViT)</h3>
<p>A ViT tokenizes an image into a grid of patches (commonly <span class="math inline">\(14 \times 14\)</span> or <span class="math inline">\(16 \times 16\)</span> pixels), flattens each into an embedding, and runs full bidirectional self-attention across all patches at every layer. Unlike causal language models, ViT attention has no causal mask, so all <span class="math inline">\(N^2\)</span> pairwise interactions are always computed — there’s no “skip the upper triangle” saving available. This makes ViTs particularly memory- and bandwidth-hungry at higher resolutions or with masked-autoencoder-style pretraining that uses large token counts. FlashAttention’s linear memory scaling directly enables training and fine-tuning ViTs at higher resolution, larger batch size, or with more patches (e.g., smaller patch size for finer-grained tokenization) than would otherwise fit in GPU memory.</p>
</section>
<section id="diffusion-models-u-net-cross-attention-and-diffusion-transformers-dit" class="level3">
<h3 class="anchored" data-anchor-id="diffusion-models-u-net-cross-attention-and-diffusion-transformers-dit" id="diffusion-models-u-net-cross-attention-and-diffusion-transformers-dit">6.2 Diffusion Models: U-Net Cross-Attention and Diffusion Transformers (DiT)</h3>
<p>Text-to-image diffusion models (e.g., Stable Diffusion) use two kinds of attention repeatedly at every denoising step: <strong>self-attention</strong> over spatial locations within the image’s latent representation, and <strong>cross-attention</strong> between the image latents (queries) and text-embedding tokens from a language encoder (keys/values). Because denoising runs for many steps (often 20–50+), even small per-step attention costs multiply substantially over a full generation. FlashAttention (via the <code>xFormers</code> memory-efficient attention op, and later native PyTorch SDPA) was one of the first widely adopted speedups for Stable Diffusion inference, commonly cited as delivering up to ~100% (2×) throughput improvements and comparable memory reductions when swapped in for the default attention implementation.</p>
<p>Newer generation architectures increasingly replace the convolutional U-Net with a <strong>Diffusion Transformer (DiT)</strong> — a pure transformer operating directly on patchified latent tokens (used in Stable Diffusion 3, PixArt, Sora-style video models, and others). DiTs push token counts even higher than U-Nets (especially for high-resolution images, e.g., up to 4K in some PixArt variants, or multi-frame video), which makes the quadratic cost of self-attention the primary scaling bottleneck of the whole model. FlashAttention is effectively a prerequisite for DiT-based models to scale to high resolutions and long video sequences at all; without it, both memory and latency would grow unacceptably with resolution.</p>
<p>Cross-attention layers in diffusion models are slightly different from self-attention in that <span class="math inline">\(Q\)</span> comes from one sequence (image latents) while <span class="math inline">\(K\)</span> and <span class="math inline">\(V\)</span> come from another (text tokens), so the two sequence lengths can differ. FlashAttention’s tiling scheme handles this transparently — the query and key/value blocks simply come from different tensors of possibly different lengths, with no algorithmic change required.</p>
</section>
<section id="multimodal-vision-language-models-clip-and-similar" class="level3">
<h3 class="anchored" data-anchor-id="multimodal-vision-language-models-clip-and-similar" id="multimodal-vision-language-models-clip-and-similar">6.3 Multimodal / Vision-Language Models (CLIP and similar)</h3>
<p>Contrastive and multimodal models like CLIP typically run a vision encoder (often a ViT) and a text encoder somewhat independently, so the direct win is the same ViT-side benefit described above. In more tightly fused vision-language architectures that interleave image and text tokens in a shared attention stack, FlashAttention’s ability to handle arbitrary (and even variable-length, padded, or block-diagonal) sequences efficiently is what makes joint image+text attention practical at scale, since the combined sequence length (image patches + text tokens) can be substantial.</p>
</section>
<section id="summary-of-the-vision-specific-payoff" class="level3">
<h3 class="anchored" data-anchor-id="summary-of-the-vision-specific-payoff" id="summary-of-the-vision-specific-payoff">6.4 Summary of the Vision-Specific Payoff</h3>
<p>Across all these architectures, the pattern is the same: image models tend to produce long token sequences (patch grids, especially at high resolution), and attention layers in vision models are frequently <em>fully bidirectional</em> with no causal mask to exploit, so the full <span class="math inline">\(N^2\)</span> cost is always paid. FlashAttention’s linear-memory, IO-aware computation is what makes it feasible to (a) increase resolution, (b) shrink patch size for finer detail, (c) increase batch size during training, and (d) run more diffusion steps or larger guidance/conditioning setups, all without hitting a memory wall or paying an outsized latency penalty.</p>
</section>
</section>
<section id="using-flashattention-in-practice" class="level2">
<h2 class="anchored" data-anchor-id="using-flashattention-in-practice" id="using-flashattention-in-practice">7. Using FlashAttention in Practice</h2>
<p>There are three common ways to get FlashAttention-style kernels into a PyTorch vision model, from most automatic to most explicit.</p>
<section id="pytorchs-built-in-scaled_dot_product_attention-easiest" class="level3">

<p>For diffusion cross-attention (differing Q and K/V sequence lengths), <code>flash-attn</code> also exposes <code>flash_attn_func</code> with independent query and key/value tensors of different lengths, or <code>flash_attn_varlen_func</code> for packed variable-length batches — handy for batching a mix of image resolutions or aspect ratios without padding waste.</p>
</section>

<section id="limitations-and-practical-considerations" class="level2">
<h2 class="anchored" data-anchor-id="limitations-and-practical-considerations" id="limitations-and-practical-considerations">9. Limitations and Practical Considerations</h2>
<ul>
<li><strong>Exact, not approximate — but only for the operation itself.</strong> FlashAttention computes the same mathematical result as standard attention (up to floating-point rounding differences from a different summation order). It does not reduce the number of tokens or approximate which tokens attend to which; if you need to reduce N itself (e.g., token pruning/merging for very high-resolution vision transformers), that’s a separate, complementary technique.</li>
<li><strong>Hardware and dtype constraints.</strong> FlashAttention kernels generally require CUDA GPUs and fp16/bf16 precision (not fp32) for the fastest paths; FlashAttention-3’s biggest gains are Hopper-only (H100/H800). On older GPUs (e.g., V100) or non-CUDA hardware, you may fall back to a slower implementation, or need vendor-specific ports (e.g., ROCm builds for AMD GPUs).</li>
<li><strong>Backward-pass recomputation cost.</strong> The backward pass recomputes attention scores rather than storing them, trading extra compute for memory savings. This is a good trade on modern GPUs (bandwidth-bound), but it does mean backward FLOPs are somewhat higher than a naive implementation that keeps the <span class="math inline">\(O(N^2)\)</span> matrix around — the net wall-clock effect is still a large win because of the bandwidth savings, but it’s not “free.”</li>
<li><strong>Custom attention variants need explicit support.</strong> Masking patterns, relative position biases, ALiBi, sliding windows, or other non-standard modifications to the attention computation need explicit kernel support (most are supported in <code>flash-attn</code> and PyTorch SDPA today, but a truly novel attention variant may not be, and may require falling back to an unfused implementation or writing a custom Triton kernel).</li>
<li><strong>Numerical differences can matter in generative pipelines.</strong> Because summation order differs from standard attention, outputs can differ at the level of floating-point rounding. For most vision and generative workloads this is inconsequential, but bit-exact reproducibility tests (e.g., comparing a saved reference output pixel-for-pixel) can be sensitive to this.</li>
<li><strong>Variable resolution / aspect ratio batching.</strong> Vision datasets often mix aspect ratios and resolutions, producing different token counts per image. Padding to a common length wastes the exact compute savings FlashAttention provides; using variable-length (“varlen”) kernels avoids this but requires restructuring the data pipeline to pack sequences.</li>
<li><strong>Diminishing returns at short sequence lengths.</strong> For small <span class="math inline">\(N\)</span> (e.g., a low-resolution ViT with few patches), the fixed overhead of FlashAttention’s block-wise looping can reduce or eliminate its advantage over a straightforward fused kernel; the benefit grows with <span class="math inline">\(N\)</span>, which is exactly the regime image models (especially high-resolution and video-diffusion ones) are pushing into.</li>
</ul>
</section>
<section id="conclusion" class="level2">
<h2 class="anchored" data-anchor-id="conclusion" id="conclusion">10. Conclusion</h2>
<p>FlashAttention did not invent a new attention mechanism — it re-engineered how the existing, well-understood softmax attention computation moves data through the GPU memory hierarchy. By tiling the computation to keep the <span class="math inline">\(O(N^2)\)</span>-sized intermediate matrices confined to fast on-chip SRAM, using an online softmax to normalize without ever materializing a full attention row at once, and recomputing rather than storing intermediates for the backward pass, it converts attention from a memory-bandwidth-bound operation into one that runs much closer to a GPU’s compute limits — all while producing numerically exact results.</p>
<p>For image models specifically, where attention is typically fully bidirectional (no causal-mask savings) and where resolution or patch-count growth causes token counts — and therefore compute and memory — to balloon quadratically, this matters enormously. FlashAttention (and its descendants, FlashAttention-2 and -3) is a large part of why modern ViTs can train at higher resolution, why Stable Diffusion-class models got faster without changing their outputs, and why Diffusion Transformer architectures generating multi-megapixel images or long video clips are computationally feasible at all. In most current PyTorch-based vision pipelines, you get these benefits automatically through <code>scaled_dot_product_attention</code>; understanding what’s happening underneath — tiling, online softmax, and IO-awareness — helps you reason about when it helps, when it doesn’t, and how to debug it when performance doesn’t match expectations.</p>
</section>
